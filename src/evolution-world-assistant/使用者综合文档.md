# Evolution World Assistant 使用者综合文档

## 1. 这套脚本到底在做什么

`Evolution World Assistant` 的目标是：

- 在你点击发送后，先拦截这一轮生成。
- 先运行你配置的"工作流"（可以并行或串行）。
- 根据工作流结果，更新角色卡绑定的世界书和 EJS 控制器。
- 把"本轮回复指令"一次性注入到当前轮。
- 然后才放行主生成。

简单理解："先更新世界状态，再让 AI 回复"

## 2. 核心概念

### 2.1 工作流（Flow）

一条工作流就是一次"上下文提取 + AI 处理 + 返回动作"的处理单元。

- 每条工作流可以单独启停。
- 每条工作流有优先级、超时、上下文楼层、提示词、采样参数。
- 多条工作流可以并行或串行执行。

### 2.2 API 配置预设

工作流不直接写 API 参数，而是引用一个"API 配置预设"。

当前支持两种模式：

- `自定义API`：你自己填 `API URL / API Key / 模型`，可点击"加载模型列表"。
- `酒馆连接器`：直接使用酒馆当前主 API + 当前模型，不需要再填 URL 和模型。

### 2.3 角色卡绑定世界书

脚本直接操作**角色卡绑定的主世界书**（而非单独的运行时世界书）。

- 如果角色卡没有绑定世界书，脚本会**自动创建并绑定**一个。
- 世界书里的受管条目：
  - `EW/Controller`（控制器）
  - `EW/Dyn/...`（动态条目，由工作流 AI 创建和管理）
- 你也可以在世界书中手动创建不带 `EW/` 前缀的条目，脚本不会碰它们。

### 2.4 楼层绑定

每次工作流提交的动态条目会**绑定到对应的聊天楼层**。

- 绑定信息存储在 `ChatMessage.data.ew_entries` 中。
- 当你删除某个楼层时，该楼层绑定的动态条目会**自动从世界书中清理**（如开启了自动清理）。
- 这对应了"这个楼层不满意就删掉、世界书改动也跟着回退"的使用场景。

### 2.5 控制器

控制器是一个 EJS 条目，用来控制哪些动态条目会在当前轮生效。

- 工作流返回 `controller_model`。
- 脚本本地渲染成完整 EJS 文本。
- 通过语法校验后，整条覆盖控制器条目。

### 2.6 阻塞式链路

"阻塞式"意思是：工作流没成功，这一轮就不发给主模型。

- 成功：提交世界书 + 注入本轮指令 + 放行生成。
- 失败：中止本轮发送，不提交半成品。

## 3. 聊天时的真实执行顺序

下面是你每次发送时的实际链路（代码里的真实行为）：

1. 监听用户发送意图（按钮点击、Enter 发送）。
2. 记录最近一次用户消息与发送上下文。
3. 在 `GENERATION_AFTER_COMMANDS` 阶段进行门控判定。
4. 只处理"普通发送"，跳过 `quiet/automatic_trigger/dry_run`。
5. 开始运行工作流管线（有总超时）。
6. 构建每条工作流请求：上下文消息（经正则+切片处理）、世界书快照、角色卡信息、全局世界书、预设信息、MVU 快照。
7. 按并行/串行调度执行工作流。
8. 校验每条响应 JSON 是否符合 `ew-flow/v1`。
9. 按优先级合并多流动作（世界书目标状态、控制器模型、回复指令）。
10. 本地渲染 EJS 控制器并做语法校验。
11. 声明式 diff 提交：对比目标状态与当前世界书，自动新建/覆盖/删除条目，一次性写入。
12. 注入本轮 `reply_instruction`（once），然后放行主生成。
13. 若中途失败，直接停止生成并提示错误。

## 4. 多流合并规则（冲突怎么判）

合并顺序与冲突策略如下：

- 排序：`priority` 高的先；同优先级按工作流顺序（后者覆盖前者）。
- 同名 `desired_entries`：高优先级覆盖低优先级；同优先级后者覆盖前者。
- `remove_entries` 与 `desired_entries` 同名冲突：同优先级或更高优先级下，`remove` 优先。
- `controller_model`：只保留最终胜出的那一份。
- `reply_instruction`：按排序拼接为单条 system 注入。

> **声明式操作**：AI 不再返回"执行什么操作"（upsert/delete/toggle），而是返回"条目最终应该是什么状态"（desired_entries）。脚本自动 diff 当前世界书来决定新建、覆盖还是切换开关。

## 5. 世界书策略与事务提交

### 5.1 目标世界书解析

脚本按以下逻辑确定要操作的世界书：

1. 读取当前角色卡的主绑定世界书。
2. 如果没有绑定世界书，自动创建并绑定一个新世界书。
3. 所有动态条目（`EW/Dyn/...`）和控制器（`EW/Controller`）直接写入这本世界书。

### 5.2 声明式 diff 提交

提交前脚本进行声明式对比：

- `desired_entries` 中的条目：不存在则新建，存在但内容/开关不同则覆盖。
- `remove_entries` 中的条目：存在则删除。
- 所有操作检查后统一 `replaceWorldbook` 一次写入，避免"只改了一半"的状态。

### 5.3 楼层绑定与清理

- 每次提交会记录哪些 `EW/Dyn/` 条目属于哪个楼层。
- 切换聊天时，如果开启了"自动清理孤立条目"，会删除已不存在的楼层所绑定的条目。

### 5.4 回滚能力

每次提交前会备份当前聊天的控制器内容。
你可以在调试页一键回滚控制器。

## 6. 配置面板完整说明

面板入口：输入框左侧魔法棒菜单中的 `Evolution 世界助手`。

### 6.1 总览

高频参数：

- `总开关`：是否启用阻塞链路。
- `调度模式`：并行或串行。
- `总超时(ms)`：整轮流程最大耗时。
- `门控时效(ms)`：发送去重窗口。

### 6.2 API配置

每个 API 预设可被多条工作流复用。

- `预设名称`：仅用于识别。
- `API模式`：`自定义API` 或 `酒馆连接器（主API）`。
- `API URL / API Key / 模型`：仅在自定义API模式下使用。

推荐理解：

- 你要"完全复用酒馆当前模型"：选 `酒馆连接器`。
- 你要"独立指定某个模型服务"：选 `自定义API`。

### 6.3 全局配置

- `动态条目前缀`（默认 `EW/Dyn/`）
- `控制器条目名`（默认 `EW/Controller`）
- `楼层绑定开关`：是否记录每个楼层的世界书修改。
- `自动清理孤立条目`：切换聊天时是否自动删除已不存在的楼层条目。
- `失败策略`：固定为失败即中止发送。

### 6.4 工作流配置

每条工作流包含五块：

- **基础信息**：名称、ID、绑定 API 预设、优先级、超时、上下文楼层。
- **生成参数**：上下文长度、最大回复、温度、Top P、惩罚项等。
- **行为参数**：名称行为、思维链请求、函数调用开关等。
- **上下文规则**：
  - **正则处理**：
    - `使用酒馆已启用的正则`：开启后聊天消息先经过酒馆当前激活的正则脚本处理（全局 + 角色卡正则）。
    - `自定义正则`：可增删自定义正则规则，每条有名称、正则表达式、替换文本。
  - **文本切片**：
    - `提取规则`：只保留 start～end 之间的文本发给 AI。
    - `排除规则`：删掉 start～end 之间的文本。
  - 处理顺序：酒馆正则 → 自定义正则 → 提取切片 → 排除切片。
- **提示词配置**：可多条、可排序、可启停、可设角色/位置/触发器。

### 6.5 调试

可直接做四类动作：

- 手动运行（不注入当前轮，仅验证流程）。
- 控制器语法校验。
- 回滚控制器。
- 查看最近运行记录、每流请求/响应摘要。

还支持：

- 导出配置（复制 JSON）。
- 导入配置（粘贴或文件导入）。

## 7. 对接协议（ew-flow/v1）

无论你用自定义模型还是连接器，最终都要输出符合协议的 JSON。

### 7.1 请求体重点

- `version/request_id/chat_id/message_id/user_input`
- `flow`（含优先级、参数、提示词条目）
- `context.messages`（已按正则和切片规则处理）
- `worldbook`（当前角色卡世界书名称和条目快照）
- `character_context`（角色名、描述、性格、场景、系统提示、越狱提示、对话示例、世界书条目）
- `world_info`（触发的世界书信息 before/after）
- `prompt_ordering`（当前工作流的提示词编排顺序快照）
- `global_worldbooks`（全局世界书条目）
- `preset_info`（当前预设名称和启用的提示词）
- `mvu.stat_data`（若可用）
- `serial_results`（串行模式下给后续流读取）

### 7.2 AI 必须返回的响应格式（核心）

**插件只接受严格符合以下 schema 的纯 JSON 对象**。不接受 markdown 代码块包裹、不接受多余文字、不接受 YAML。

#### 完整字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | `"ew-flow/v1"` | ✅ | 固定值，协议版本 |
| `flow_id` | `string` | ✅ | **必须与请求中 `flow.id` 完全一致** |
| `status` | `"ok"` | ✅ | 固定值，必须是 `"ok"` |
| `priority` | `number` | ✅ | **必须与请求中 `flow.priority` 一致** |
| `reply_instruction` | `string` | ❌ | 本轮回复指令，注入后会作为 system 消息影响主模型回复。留空则不注入。 |
| `operations.worldbook.desired_entries` | `array` | ✅ | 目标状态条目（允许空数组 `[]`）。每个对象：`{name, content, enabled}` |
| `operations.worldbook.remove_entries` | `array` | ✅ | 要删除的条目（允许空数组 `[]`）。每个对象：`{name}` |
| `operations.controller_model` | `object` | ✅ | **控制器模型（必填）**。所有启用的工作流合并后必须至少有一份，否则脚本报错。结构见下方。 |
| `diagnostics.trace_id` | `string` | ❌ | 调试追踪 ID，可选 |

#### `desired_entries` 条目格式

```json
{
  "name": "EW/Dyn/条目名",
  "content": "条目文本内容",
  "enabled": true
}
```

- `name`：**必须以受管前缀开头**（默认 `EW/Dyn/`），否则脚本不会处理。
- `content`：条目文本，会直接写入世界书。
- `enabled`：`true` 表示条目启用，`false` 表示条目存在但禁用（不会被注入上下文）。

#### `controller_model` 格式与 EJS 渲染机制

AI 返回的 `controller_model` **并不是直接使用的配置**，而是会被脚本**自动渲染成 EJS 模板代码**，写入世界书的 `EW/Controller` 条目。SillyTavern 在每轮生成时会执行这段 EJS，动态决定哪些世界书条目的内容会被注入上下文。

```json
{
  "template_id": "entry_selector_v1",
  "variables": [
    { "name": "goodwill", "path": "stat_data.关系.好感", "default": 0 }
  ],
  "rules": [
    { "when": "goodwill >= 50", "include_entries": ["EW/Dyn/阶段2剧情"] }
  ],
  "fallback_entries": ["EW/Dyn/阶段1剧情"]
}
```

**上面的 JSON 会被自动渲染成以下 EJS 模板，写入 `EW/Controller` 条目：**

```ejs
<%_
if (typeof goodwill === 'undefined') var goodwill = getvar('stat_data.关系.好感', { defaults: 0 });
_%>

<%_ if (goodwill >= 50) { _%>
<%- await getwi(null, 'EW/Dyn/阶段2剧情') %>
<%_ } else { _%>
<%- await getwi(null, 'EW/Dyn/阶段1剧情') %>
<%_ } _%>
```

各字段含义：

- `template_id`：当前固定为 `"entry_selector_v1"`（唯一支持的模板）。
- `variables`：变量声明列表。每个变量从 `stat_data` 中通过 `getvar()` 读取值。
  - `name`：变量名（会成为 EJS 中的 JS 变量名）。
  - `path`：读取路径（如 `stat_data.关系.好感`）。
  - `default`：路径不存在时的默认值。
- `rules`：条件分支数组，渲染成 `if / else if` 链。
  - `when`：JS 条件表达式（使用 variables 中声明的变量名）。
  - `include_entries`：条件为真时，通过 `getwi(null, '条目名')` 引入的世界书条目。
- `fallback_entries`：所有 rules 都不满足时引入的条目（渲染为 `else` 分支）。

> **核心理解**：`getwi(null, 'EW/Dyn/条目名')` 是 SillyTavern 的 EJS 函数，它会把对应世界书条目的文本内容注入到上下文中。控制器通过条件分支决定"这一轮哪些条目的内容会被 AI 看到"。

### 7.3 最小可用响应模板

所有工作流**合并后必须包含 `controller_model`**（至少一个工作流提供），否则合并阶段会报错。

最小模板（不修改世界书、不注入回复指令，但提供空控制器）：

```json
{
  "version": "ew-flow/v1",
  "flow_id": "（原样返回请求中 flow.id）",
  "status": "ok",
  "priority": 100,
  "reply_instruction": "",
  "operations": {
    "worldbook": {
      "desired_entries": [],
      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "variables": [],
      "rules": [],
      "fallback_entries": []
    }
  }
}
```

> **关键要求**：`operations.worldbook` 和 `operations.controller_model` 都必须存在。如果 `fallback_entries` 和 `rules` 都为空数组，控制器不会引入任何条目（相当于"什么都不做"）。

### 7.4 完整响应示例

```json
{
  "version": "ew-flow/v1",
  "flow_id": "flow_plot",
  "status": "ok",
  "priority": 100,
  "reply_instruction": "请以悬疑语气继续推进情节。",
  "operations": {
    "worldbook": {
      "desired_entries": [
        { "name": "EW/Dyn/线索A", "content": "主角发现了一封密信", "enabled": true },
        { "name": "EW/Dyn/暗线", "content": "幕后黑手与药店有关", "enabled": false }
      ],
      "remove_entries": [
        { "name": "EW/Dyn/过期线索" }
      ]
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "variables": [
        { "name": "goodwill", "path": "stat_data.关系.好感", "default": 0 }
      ],
      "rules": [
        { "when": "goodwill >= 50", "include_entries": ["EW/Dyn/阶段2"] }
      ],
      "fallback_entries": ["EW/Dyn/阶段1"]
    }
  },
  "diagnostics": { "trace_id": "abc-123" }
}
```

### 7.5 常见格式错误与排错

| 错误现象 | 原因 | 解决方案 |
|---------|------|---------|
| `schema 非法` | 返回的 JSON 缺少必填字段 | 确保 `version`、`flow_id`、`status`、`priority`、`operations.worldbook` 都存在 |
| `schema 非法` | `status` 不是 `"ok"` | 必须固定为 `"ok"` 字符串 |
| `schema 非法` | `flow_id` 与请求不一致 | 必须原样返回 `flow.id` |
| `无法解析 JSON` | AI 返回了 markdown 代码块 | 提示词中明确要求"纯 JSON，不要代码块" |
| `无法解析 JSON` | AI 返回了多余的解释文本 | 提示词中明确要求"只输出 JSON" |
| `EJS 语法错误` | `controller_model` 渲染后的模板不合法 | 先简化规则，在调试页做"控制器语法校验" |

### 7.6 提示词编排系统（Prompt Ordering）

工作流使用酒馆预设风格的提示词编排系统。你可以在配置面板中拖拽排列以下组件的顺序：

#### 系统标记（Marker，自动填充）

| 标记 | 内容来源 |
|------|---------|
| `main` | 角色卡的 System Prompt |
| `worldInfoBefore` | 触发的世界书信息（前置） |
| `personaDescription` | 用户人设描述 |
| `charDescription` | 角色描述 |
| `charPersonality` | 角色性格 |
| `scenario` | 场景 |
| `enhanceDefinitions` | 增强定义 |
| `worldInfoAfter` | 触发的世界书信息（后置） |
| `dialogueExamples` | 对话示例 |
| `chatHistory` | 聊天记录 |
| `postHistoryInstructions` | 角色卡的 Jailbreak / Post-History 指令 |

#### 自定义提示词（Prompt，用户编辑）

- 可以设置 `角色`（system/user/assistant）、`插入位置`（相对/聊天中）、`注入深度`。
- `相对`：按在编排列表中的前后顺序插入。
- `聊天中`：插入到聊天记录中，按 `注入深度` 从末尾向前计算位置（深度 0 = 最后一条后，深度 1 = 倒数第一条前，以此类推）。

#### 编排顺序如何影响 AI 收到的消息

脚本会按照你在面板中排列的顺序，从上到下依次组装 `messages` 数组发给 AI。禁用（关闭开关）的条目会被跳过。最终 AI 收到的消息序列就是你编排的顺序。

## 9. 常见问题与排错

### 9.1 发送后没触发脚本

优先检查：

- 总开关是否开启。
- 触发是否是普通发送（不是 quiet / 自动触发）。
- 门控时效是否太短导致上下文失效。

### 9.2 报 schema 非法

说明返回 JSON 不符合 `ew-flow/v1`。

重点检查：

- `status` 是否为 `ok`
- `operations.worldbook` 字段是否存在
- `flow_id/priority` 是否可解析

### 9.3 报 EJS 语法错误

说明 `controller_model` 渲染后的模板不合法。

建议：

- 先简化规则，确认最小模板可过。
- 在调试页做"控制器语法校验"。

### 9.4 自定义API模式无法运行

重点检查：

- `API URL` 是否可用。
- `模型` 是否已填写。
- 是否能通过"加载模型列表"拿到模型。

### 9.5 世界书看起来没更新

重点检查：

- 动态条目名是否在受管范围（默认 `EW/Dyn/`）。
- 是否在 `remove_entries` 中被删除。
- 是否被控制器规则过滤掉（条目在书里但未启用）。

## 10. 对深入研究者的模块地图

如果你想继续深挖源码，按这个顺序读：

1. `runtime/events.ts`：什么时候触发。
2. `runtime/pipeline.ts`：完整主流程。
3. `runtime/dispatcher.ts`：多流调度与 API 调用。
4. `runtime/context-builder.ts`：上下文构建（正则处理、切片、角色卡信息收集）。
5. `runtime/merger.ts`：冲突合并规则。
6. `runtime/transaction.ts`：声明式 diff 事务提交。
7. `runtime/worldbook-runtime.ts`：角色卡世界书解析与自动创建。
8. `runtime/floor-binding.ts`：楼层绑定与孤立条目清理。
9. `runtime/controller-renderer.ts`：控制器渲染与语法校验。
10. `ui/App.vue` + `ui/components/*.vue`：面板结构和字段绑定。

## 11. 安全与稳定性建议

- API Key 只给最小权限，尽量使用代理层。
- 总超时建议大于所有启用流最大超时。
- 串行模式用于"后流依赖前流结果"的场景。
- 并行模式用于"互不依赖、追求速度"的场景。
- 大改前先导出配置，必要时可快速恢复。

## 12. 当前实现边界（请知悉）

- 失败策略当前固定：失败即中止发送。
- 控制器模板当前固定：`entry_selector_v1`。
- `MVU` 快照是"可用则读取，不可用则为空对象"。
- 旧版"直接 HTTP 工作流接口"仅做兼容，不作为当前 UI 主路径。

---

如果你要继续扩展，最值得先做的是：

- 增加模板族（不仅 `entry_selector_v1`）。
- 为不同工作流增加更明确的输出约束模板。
- 做一组可复用的"工作流预设包"（抽取、角色状态、事件推进）。
