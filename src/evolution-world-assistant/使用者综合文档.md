# Evolution World Assistant 使用者综合文档

## 1. 这套脚本到底在做什么

`Evolution World Assistant` 的目标是：

- 在你点击发送后，先拦截这一轮生成。
- 先运行你配置的“工作流”（可以并行或串行）。
- 根据工作流结果，更新聊天级运行时世界书和 EJS 控制器。
- 把“本轮回复指令”一次性注入到当前轮。
- 然后才放行主生成。

简单理解：“先更新世界状态，再让 AI 回复”

## 2. 核心概念（先建立共同语言）

### 2.1 工作流（Flow）

一条工作流就是一次“上下文提取 + AI 处理 + 返回动作”的处理单元。

- 每条工作流可以单独启停。
- 每条工作流有优先级、超时、上下文楼层、提示词、采样参数。
- 多条工作流可以并行或串行执行。

### 2.2 API 配置预设

工作流不直接写 API 参数，而是引用一个“API 配置预设”。

当前支持两种模式：

- `自定义API`：你自己填 `API URL / API Key / 模型`，可点击“加载模型列表”。
- `酒馆连接器`：直接使用酒馆当前主 API + 当前模型，不需要再填 URL 和模型。

### 2.3 聊天级运行时世界书

脚本不会把所有聊天混在一起，而是按聊天隔离。

- 每个聊天对应一本运行时世界书（默认名：`EW_RUNTIME::<chat_id>`）。
- 世界书里有固定条目：
- `EW/Controller`（控制器）
- `EW/Meta`（元数据）
- `EW/Dyn/...`（动态条目）

### 2.4 控制器

控制器是一个 EJS 条目，用来控制哪些动态条目会在当前轮生效。

- 工作流返回 `controller_model`。
- 脚本本地渲染成完整 EJS 文本。
- 通过语法校验后，整条覆盖控制器条目。

### 2.5 阻塞式链路

“阻塞式”意思是：工作流没成功，这一轮就不发给主模型。

- 成功：提交世界书 + 注入本轮指令 + 放行生成。
- 失败：中止本轮发送，不提交半成品。

## 3. 聊天时的真实执行顺序

下面是你每次发送时的实际链路（代码里的真实行为）：

1. 监听用户发送意图（按钮点击、Enter 发送）。
2. 记录最近一次用户消息与发送上下文。
3. 在 `GENERATION_AFTER_COMMANDS` 阶段进行门控判定。
4. 只处理“普通发送”，跳过 `quiet/automatic_trigger/dry_run`。
5. 开始运行工作流管线（有总超时）。
6. 构建每条工作流请求：上下文消息、世界书快照、MVU 快照等。
7. 按并行/串行调度执行工作流。
8. 校验每条响应 JSON 是否符合 `ew-flow/v1`。
9. 按优先级合并多流动作（世界书操作、控制器模型、回复指令）。
10. 本地渲染 EJS 控制器并做语法校验。
11. 事务提交：世界书变更 + 控制器重写一次性写入。
12. 注入本轮 `reply_instruction`（once），然后放行主生成。
13. 若中途失败，直接停止生成并提示错误。

## 4. 多流合并规则（冲突怎么判）

合并顺序与冲突策略如下：

- 排序：`priority` 高的先；同优先级按工作流顺序（后者覆盖前者）。
- 同名 `upsert`：高优先级覆盖低优先级；同优先级后者覆盖前者。
- `delete` 与 `upsert` 同名冲突：同优先级下 `delete` 优先。
- `toggle` 目标不存在：直接报错（整轮失败）。
- `controller_model`：只保留最终胜出的那一份。
- `reply_instruction`：按排序拼接为单条 system 注入。

## 5. 世界书策略与事务提交

### 5.1 自动发现/创建逻辑

脚本会按下面顺序找运行时世界书：

1. 先看当前聊天已绑定世界书是否带有效元数据标记。
2. 再按前缀扫描候选世界书（受“最大扫描世界书数”限制）。
3. 都找不到就新建并绑定。

### 5.2 为什么说是“事务化”

提交前会先做预检查（如条目名是否属于受管前缀）。
通过后才统一 `replaceWorldbook` 一次写入。

这避免了“只改了一半”的状态。

### 5.3 回滚能力

每次提交前会备份当前聊天的控制器内容。
你可以在调试页一键回滚控制器。

## 6. 配置面板完整说明

面板入口：输入框左侧魔法棒菜单中的 `Evolution 世界助手`。

### 6.1 总览

高频参数：

- `总开关`：是否启用阻塞链路。
- `调度模式`：并行或串行。
- `总超时(ms)`：整轮流程最大耗时。
- `门控时效(ms)`：发送去重窗口。

### 6.2 API配置

每个 API 预设可被多条工作流复用。

字段说明：

- `预设名称`：仅用于识别。
- `API模式`：`自定义API` 或 `酒馆连接器（主API）`。
- `API URL`：仅在自定义API模式下使用。
- `API Key`：仅在自定义API模式下使用。
- `模型`：仅在自定义API模式下使用。
- `加载模型列表`：根据 URL/Key 拉取可选模型。

推荐理解：

- 你要“完全复用酒馆当前模型”：选 `酒馆连接器`。
- 你要“独立指定某个模型服务”：选 `自定义API`。

### 6.3 全局配置

基础：

- `运行时世界书前缀`
- `动态条目前缀`
- `控制器条目名`
- `元数据条目名`

高级：

- `元数据标记`
- `最大扫描世界书数`
- `失败策略（固定为失败即中止发送）`

### 6.4 工作流配置

每条工作流包含四块：

- 基础信息：名称、ID、绑定 API 预设、优先级、超时、上下文楼层。
- 生成参数：上下文长度、最大回复、温度、Top P、惩罚项等。
- 行为参数：名称行为、思维链请求、函数调用开关等。
- 提示词配置：可多条、可排序、可启停、可设角色/位置/触发器。

提示词条目关键点：

- `位置`：`相对` 或 `聊天中`。
- `触发器`：`all/send/continue/regenerate/quiet/manual`。
- 一条提示词就是一段独立职责，建议不要把所有约束塞一条。

上下文规则：

- `提取规则`：按 `start/end` 抽取片段。
- `排除规则`：按 `start/end` 去掉片段。

### 6.5 调试

可直接做四类动作：

- 手动运行（不注入当前轮，仅验证流程）。
- 控制器语法校验。
- 回滚控制器。
- 查看最近运行记录、每流请求/响应摘要。

还支持：

- 导出配置（复制 JSON）。
- 导入配置（粘贴或文件导入）。

## 7. 对接协议（ew-flow/v1）

无论你用自定义模型还是连接器，最终都要输出符合协议的 JSON。

### 7.1 请求体重点

- `version/request_id/chat_id/message_id/user_input`
- `flow`（含优先级、参数、提示词条目）
- `context.messages`（已按楼层和规则处理）
- `worldbook.entries`（当前运行时世界书快照）
- `mvu.stat_data`（若可用）
- `serial_results`（串行模式下给后续流读取）

### 7.2 响应体重点

- `status` 必须是 `ok`
- `operations.worldbook` 可包含 `upsert/delete/toggle`
- `operations.controller_model` 可选，但合并后必须至少有一份
- `reply_instruction` 用于本轮注入

简版示例：

```json
{
  "version": "ew-flow/v1",
  "flow_id": "flow_plot",
  "status": "ok",
  "priority": 100,
  "reply_instruction": "请以悬疑语气继续推进情节。",
  "operations": {
    "worldbook": {
      "upsert_entries": [
        { "name": "EW/Dyn/线索A", "content": "...", "enabled": true }
      ],
      "delete_entries": [],
      "toggle_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "variables": [
        { "name": "goodwill", "path": "stat_data.关系.好感", "default": 0 }
      ],
      "rules": [
        { "when": "goodwill >= 50", "include_entries": ["EW/Dyn/阶段2"] }
      ],
      "fallback_entries": ["EW/Dyn/阶段1"]
    }
  },
  "diagnostics": { "trace_id": "abc-123" }
}
```

## 8. 新手推荐配置路径（10 分钟跑通）

1. 打开魔法棒菜单，进入 `Evolution 世界助手`。
2. 在“总览”打开总开关，调度模式先用并行。
3. 在“API配置”新增 1 个预设：

- 如果想复用酒馆当前模型，选 `酒馆连接器`。
- 如果想指定独立模型，选 `自定义API` 并填 URL/Key/模型。

1. 在“工作流配置”保留 1 条工作流，绑定刚才的 API 预设。
2. 给工作流写 1~2 条最小提示词（例如“输出必须是 ew-flow/v1 JSON”）。
3. 在“调试”里先手动运行，确认返回可过 schema。
4. 开始正常聊天，观察“最近请求/响应摘要”。

## 9. 常见问题与排错

### 9.1 发送后没触发脚本

优先检查：

- 总开关是否开启。
- 触发是否是普通发送（不是 quiet / 自动触发）。
- 门控时效是否太短导致上下文失效。

### 9.2 报 schema 非法

说明返回 JSON 不符合 `ew-flow/v1`。

重点检查：

- `status` 是否为 `ok`
- `operations.worldbook` 字段是否存在
- `flow_id/priority` 是否可解析

### 9.3 报 EJS 语法错误

说明 `controller_model` 渲染后的模板不合法。

建议：

- 先简化规则，确认最小模板可过。
- 在调试页做“控制器语法校验”。

### 9.4 自定义API模式无法运行

重点检查：

- `API URL` 是否可用。
- `模型` 是否已填写。
- 是否能通过“加载模型列表”拿到模型。

### 9.5 世界书看起来没更新

重点检查：

- 动态条目名是否在受管范围（默认 `EW/Dyn/`）。
- 是否返回了 `delete` 把条目删掉。
- 是否被控制器规则过滤掉（条目在书里但未启用）。

## 10. 对深入研究者的模块地图

如果你想继续深挖源码，按这个顺序读：

1. `runtime/events.ts`：什么时候触发。
2. `runtime/pipeline.ts`：完整主流程。
3. `runtime/dispatcher.ts`：多流调度与 API 调用。
4. `runtime/merger.ts`：冲突合并规则。
5. `runtime/transaction.ts`：事务提交与保护。
6. `runtime/worldbook-runtime.ts`：聊天级世界书自动发现/创建。
7. `runtime/controller-renderer.ts`：控制器渲染与语法校验。
8. `ui/App.vue` + `ui/components/*.vue`：面板结构和字段绑定。

## 11. 安全与稳定性建议

- API Key 只给最小权限，尽量使用代理层。
- 总超时建议大于所有启用流最大超时。
- 串行模式用于“后流依赖前流结果”的场景。
- 并行模式用于“互不依赖、追求速度”的场景。
- 大改前先导出配置，必要时可快速恢复。

## 12. 当前实现边界（请知悉）

- 失败策略当前固定：失败即中止发送。
- 控制器模板当前固定：`entry_selector_v1`。
- `MVU` 快照是“可用则读取，不可用则为空对象”。
- 旧版“直接 HTTP 工作流接口”仅做兼容，不作为当前 UI 主路径。

---

如果你要继续扩展，最值得先做的是：

- 增加模板族（不仅 `entry_selector_v1`）。
- 为不同工作流增加更明确的输出约束模板。
- 做一组可复用的“工作流预设包”（抽取、角色状态、事件推进）。
