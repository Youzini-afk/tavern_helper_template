# Evolution World Assistant 使用者综合文档

## 1. 这套脚本在做什么

**"先更新世界状态，再让 AI 回复。"**

1. 拦截发送 → 运行工作流 AI → 更新世界书 + 控制器 → 注入本轮指令 → 放行主生成。

```text
发送消息 → 拦截 → 工作流AI分析 → 返回结果
  ├── EW/Dyn/ 动态条目 → 写入世界书（始终禁用/红灯）
  ├── controller_model → 插件自动编译为 EJS → 写入 EW/Controller
  └── reply_instruction → 注入本轮 → 放行生成
```

### 两层架构

- **EW/Dyn/ 条目 = 数据仓库**：存储动态信息，始终保持**禁用**状态（红灯）。
- **EW/Controller = 选择器**：通过 EJS `getwi()` 按需拉取指定 Dyn 条目注入上下文。

好处：世界书积累大量条目，每轮只注入相关的几条，节省 token。

> [!IMPORTANT]
> AI **不需要写 EJS 代码**。AI 只返回结构化的 `controller_model` JSON 字段，插件自动编译为 EJS。

## 2. 核心概念

### 工作流（Flow）

一次"上下文提取 + AI 处理 + 返回动作"的处理单元。可单独启停，有优先级/超时/上下文楼层/提示词/采样参数。多条可并行或串行执行。

### 触发时机

| 时机 | 说明 |
| ---- | ---- |
| `before_reply` | 发送消息前拦截，工作流完成后放行主生成（默认） |
| `after_reply` | AI 回复完成后运行，用于后处理/状态更新 |
| `manual` | 调试面板手动触发 |

> 触发类型：`continue`/`regenerate`/`swipe` 时自动取最后一条用户消息作为 `user_input`。

### API 配置预设

- **自定义 API**：自填 URL/Key/模型。
- **酒馆连接器**：复用酒馆当前主 API + 模型。

### 世界书

脚本操作角色卡绑定的主世界书。没有绑定时自动创建。受管条目：`EW/Controller`（启用）、`EW/Dyn/*`（禁用）。

### 内置世界信息引擎

EW 拥有独立的世界信息引擎（`worldinfo-engine.ts`），在工作流提示词组装时自行解析世界书，**不依赖** ST 预拼接的 worldInfoBefore/After 文本。

引擎复制了 ST 的激活逻辑：常量条目、装饰器（`@@activate`/`@@dont_activate`/`@@only_preload`）、关键词匹配（正则/大小写/全词）、次关键词逻辑（AND_ANY/AND_ALL/NOT_ANY/NOT_ALL）、概率过滤、互斥组等。

### 内置 EJS 引擎

`ejs-internal.ts` 提供完整的 EJS 运行时，支持：

- `getwi(worldbook, entryName)` — worldbook-aware 的条目拉取
- `getvar(path, { scope, defaults })` — 四级变量读取（global/local/message/cache）
- `setvar(path, value, { scope })` — 内存态变量写入（仅影响当前渲染 pass）
- `activewi(world, entry, force)` — 上下文内条目激活
- `getChatMessages(start, end, role)` / `getChatMessage(index, role)` — 聊天消息访问
- `{{mustache}}` 宏替换（`{{user}}`、`{{char}}`、`{{lastUserMessage}}` 等）

### 楼层绑定与快照

每次工作流提交的动态条目绑定到对应聊天楼层。删除楼层时自动回滚世界书状态。

**快照存储方式**（高级设置可切换）：

| 模式 | 存储位置 | 适用场景 |
| ---- | -------- | -------- |
| 消息数据 | 聊天消息的 data 字段 | 短聊天、简单场景 |
| **服务器文件**（默认） | ST 服务器 `data/{user}/files/` | 长聊天、数据量大 |

- 切换模式后点击「同步快照」按钮可迁移已有数据。
- 聊天切换时自动 purge + restore：删楼回滚、新聊天干净、切回旧聊天恢复。

### 控制器

工作流返回 `controller_model` → 插件自动渲染成 EJS → 语法校验（`await` 关键字安全剥离后检查）→ 写入 `EW/Controller`。

### 失败策略

| 策略 | 行为 |
| ---- | ---- |
| `stop_generation` | 中止发送 |
| `continue_generation` | 失败但继续生成 |
| `retry_once` | 重试一次 |
| `notify_only` | 仅通知 |
| `allow_partial_success` | 有一条流成功就继续 |

## 3. 执行顺序

1. 监听发送 → 门控判定（跳过 quiet/automatic_trigger/dry_run）
2. 隐藏消息检查 → 收集被忽略 WI 条目内容（MVU 过滤用）
3. 构建请求上下文（内置 WI 引擎解析 → MVU 过滤 → 正则 → 切片 → 提示词组装）
4. 并行/串行调度 → 校验响应 → 按优先级合并
5. 编译 controller_model → EJS 模板 → 语法校验
6. 声明式 diff 提交世界书（Dyn 条目强制 `enabled=false`）→ 注入 reply_instruction → 放行生成

## 4. 多流合并规则

- 排序：`priority` 高的先；同优先级后者覆盖。
- 同名 `desired_entries`：高优先级覆盖。
- `remove_entries` 与 `desired_entries` 冲突：`remove` 优先。
- `controller_model`：保留最终胜出的。
- `reply_instruction`：按顺序拼接。

## 5. 配置面板

面板入口：输入框左侧魔法棒 → `Evolution 世界助手`。

### 总览

总开关、调度模式（并行/串行）、总超时、门控时效。

### API 配置

每个预设可被多条工作流复用。`自定义 API`（填 URL/Key/模型）或 `酒馆连接器`（复用当前主 API）。

### 全局配置

动态条目前缀、控制器条目名、楼层绑定开关、自动清理孤立条目。

### 高级配置

- 失败策略
- **快照存储方式**：消息数据 / 服务器文件 + 同步快照按钮

### 工作流配置

基础信息、生成参数、行为参数、上下文规则（正则 + 切片）、提示词编排（拖拽排列）。

### 隐藏消息

- **AI 隐藏**（`hide_last_n`）：保留最近 N 条可见，更旧的标记 `is_system=true`。
- **楼层限制器**（`limiter_count`）：只渲染最近 M 条，不影响 AI 上下文。

### 调试

手动运行、控制器校验、回滚控制器、重 Roll 当前楼层、请求/响应摘要、导入/导出配置。

## 6. 对接协议（ew-flow/v1）

### 请求体关键字段

| 字段 | 说明 |
| ---- | ---- |
| `version` | `"ew-flow/v1"` |
| `flow` | 工作流配置快照 |
| `user_input` | 用户输入的消息 |
| `trigger` | 触发信息（timing/source/generation_type/message_id） |
| `context` | 上下文元信息（turns/extract_rules/exclude_rules） |
| `worldbook` | 世界书名称（用于写入操作定位） |
| `serial_results` | 串行前置流结果 |

> **注意**：角色卡字段、世界信息、聊天记录、自定义提示词均通过 **提示词编排** 注入，不在 JSON 请求体中。

### AI 响应格式

**纯 JSON，不要代码块、不要额外文字。**

```json
{
  "version": "ew-flow/v1",
  "flow_id": "原样返回请求中 flow.id",
  "status": "ok",
  "priority": 100,
  "reply_instruction": "",
  "operations": {
    "worldbook": {
      "desired_entries": [
        { "name": "EW/Dyn/条目名", "content": "内容", "enabled": true }
      ],
      "remove_entries": []
    },
    "controller_model": {
      "template_id": "entry_selector_v1",
      "fallback_entries": ["主时间线"],
      "char_detection": {
        "alias_map": { "角色A": "角色A", "角色B": "角色B" },
        "scene_var": "stat_data.在场人物",
        "scan_messages": 1,
        "entry_patterns": ["{name}", "{name}_记忆"]
      },
      "rules": [],
      "variables": [],
      "set_variables": [],
      "activate_entries": [],
      "inject_text": [],
      "for_each": [],
      "decorators": [],
      "skip_floor_zero": false
    }
  }
}
```

**必填字段**：`version`、`flow_id`、`status`、`priority`、`operations.worldbook`。
**推荐提供**：`controller_model`（不提供则控制器为空）。

### controller_model 渲染机制

AI 返回 `controller_model` JSON → 插件自动编译为 EJS 模板 → 写入 `EW/Controller`。

| 字段 | 编译结果 |
| ---- | -------- |
| `fallback_entries` | 直接 `getwi()` 拉取的兜底条目 |
| `char_detection` | 基于别名映射 + 消息扫描的角色检测，`for...of` 循环拉取对应条目 |
| `rules` | `if/else` 链，条件满足时 `getwi()` 引入条目 |
| `variables` | 从变量中读取值（`getvar()`） |
| `set_variables` | 渲染时设置变量（`setvar()`） |
| `activate_entries` | 激活指定条目（`activewi()`） |
| `inject_text` | 直接注入的纯文本 |
| `for_each` | 遍历变量列表拉取条目 |
| `decorators` | EJS 装饰器注释（如 `@@activate` 等） |
| `skip_floor_zero` | 第 0 楼（开场白）时跳过 Controller |

### 常见错误

| 错误 | 解决 |
| ---- | ---- |
| schema 非法 | 检查必填字段是否齐全、`status` 是否为 `"ok"` |
| 无法解析 JSON | 提示词要求"只输出 JSON，不要代码块" |
| EJS 语法错误 | 简化 controller_model 规则，用调试页校验 |
| workflow already processing | 刷新页面清除残留状态锁 |

## 7. MVU 兼容层

当角色卡同时使用了 **MagVarUpdate（MVU）** 插件时，MVU 会向世界书和扩展提示词注入自己的变量更新/状态占位内容。这些内容对 EW 工作流是干扰项——如果不过滤，工作流 AI 会看到大量无关的变量更新指令，浪费 token 并影响输出质量。

`mvu-compat.ts` 提供三级过滤策略：

### 第一级：世界书条目过滤

在 `worldinfo-engine` 解析世界书时，名称或 comment 包含 MVU 标记的条目（`[mvu_update]`、`[mvu_plot]`、`[initvar]`）会被忽略，不参与工作流上下文构建。

同时对条目**内容**做启发式检测——即使条目名称没有 MVU 标记，如果内容是典型的 MVU 产物（如 `<status_current_variable>`、变量输出格式、变量更新规则等），也会被过滤。

### 第二级：扩展提示词内容剥离

酒馆的 `extension_prompts` 中可能包含 MVU 注入的 XML 块（`<updatevariable>...</updatevariable>`、`<StatusPlaceHolderImpl/>`、`<status_current_variable>...</status_current_variable>`）。EW 在组装工作流提示词时，会用正则从扩展提示词中**剥离**这些 MVU 片段，只保留有用的内容。

### 第三级：被忽略条目内容去重

某些情况下，MVU 的世界书条目内容可能被酒馆的 WI 引擎提前渲染并嵌入到其他提示词中。EW 在工作流启动时会收集所有被过滤的世界书条目的原始内容（`collectIgnoredWorldInfoContents`），然后在扩展提示词中做子串匹配剥离——如果某段提示词完整包含了被忽略条目的内容，该内容会被移除。

### 用户注意事项

- **无需手动配置**：MVU 过滤全自动，开箱即用。
- **不影响正常内容**：过滤规则经过精心设计，只针对 MVU 特征模式，不会误伤普通世界书条目或自定义提示词。
- **兼容性**：即使角色卡没有使用 MVU，过滤层也不会产生任何副作用。
